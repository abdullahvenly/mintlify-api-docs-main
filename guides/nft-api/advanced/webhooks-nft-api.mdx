**Real-time Notifications:** Set up webhooks to receive instant notifications for contract creations, token-type creations, and token mints - both successful and failed events.

**Time required:** ~10 minutes
</Info>

## What are Webhooks?

Webhooks allow you to configure your endpoint URL and receive real-time notifications for all your NFT assets. Instead of constantly polling our API for status updates, webhooks push notifications directly to your server when events occur.

<CardGroup cols={2}>
  <Card title="Real-time Updates" icon="bolt">

  </Card>
  <Card title="Reliable Delivery" icon="shield-check">

  </Card>
  <Card title="Comprehensive Coverage" icon="list-check">

  </Card>
  <Card title="Easy Integration" icon="plug">

  </Card>

## Supported Events

The NFT-API webhooks notify you about the following events:

<Tabs>
  <Tab title="Success Events">
    - **Contract Creation Succeeded**
    - **Token-type Creation Succeeded**
    - **Token Mint Succeeded**
  </Tab>
  <Tab title="Failure Events">
    - **Contract Creation Failed**
    - **Token-type Creation Failed**
    - **Token Mint Failed**
  </Tab>
</Tabs>

<Note>
Look for `eventType` and `result.status` in the payloads you receive to identify the specific event and outcome.
</Note>

## Retry Mechanism

<Warning>
**Reliability Guarantee:** If your endpoint URL is down, Venly will attempt to resend the notification message **5 times** with **10-second intervals** to ensure delivery.
</Warning>

### Backup Status Checking

If your endpoint is down for more than **50 seconds** and you miss the webhook notification, you can use these status check endpoints:

<AccordionGroup>
  <Accordion title="Check Contract Status" icon="file-contract">
    Use the contract deployment `id` from your creation response:
    
    ```javascript
    const response = await fetch(`/api/v3/erc1155/contracts/deployments/${deploymentId}`, {
      headers: { 'Authorization': 'Bearer YOUR_API_KEY' }
    });
    ```
    
    [API Reference](https://docs.venly.io/reference/getcontract_1-1)
  </Accordion>
  
  <Accordion title="Check Token-type Status" icon="shapes">
    Use the token-type creation `id` from your creation response:
    
    ```javascript
    const response = await fetch(`/api/v3/erc1155/token-types/creations/${creationId}`, {
      headers: { 'Authorization': 'Bearer YOUR_API_KEY' }
    });
    ```
    
    [API Reference](https://docs.venly.io/reference/gettokentype)
  </Accordion>
  
  <Accordion title="Check Token Mint Status" icon="coins">
    Use the mint `id` from your minting response:
    
    ```javascript

    ```
    
    [API Reference](https://docs.venly.io/reference/gettokenmint)
  </Accordion>
</AccordionGroup>

## How to Configure Webhooks

<Steps>
  <Step title="Access Developer Portal" icon="external-link">
    Navigate to the [Developer Portal](https://portal.venly.io/) and sign in to your account
  </Step>
  
  <Step title="Select NFT API Configuration" icon="gear">
    Under **Configuration**, click on **NFT API** to access webhook settings
  </Step>
  
  <Step title="Create New Webhook" icon="plus">
    Scroll down and click on **Create webhook** to start the setup process
  </Step>
  
  <Step title="Configure Webhook Details" icon="form">
    Fill in the webhook configuration form with your endpoint details
  </Step>
  
  <Step title="Test and Activate" icon="check">
    Save your webhook configuration and test the integration
  </Step>
</Steps>

### Webhook Configuration Form

When creating a webhook, you'll need to provide:

| Field | Description | Required |
|-------|-------------|----------|
| **Description** | Any descriptive name for the webhook | ✅ |
| **Endpoint (POST)** | Your server endpoint URL that accepts POST requests | ✅ |
| **Type** | Authentication type (Basic Auth, API Key, etc.) | ✅ |
| **Username/Header name** | Authentication username or custom header name | ✅ |
| **Password/API key** | Authentication password or API key value | ✅ |

<Tip>
**Authentication Setup:** You need to set up authentication on your endpoint to verify that calls are coming from Venly. This ensures security and prevents unauthorized webhook calls.
</Tip>

## Implementing Webhook Endpoints

<CodeGroup>
```javascript Node.js/Express
const express = require('express');
const app = express();

app.use(express.json());

// Webhook endpoint
app.post('/webhooks/nft-api', (req, res) => {
  const { eventType, result } = req.body;
  
  console.log(`Received webhook: ${eventType}`);
  console.log(`Status: ${result.status}`);
  
  // Process the webhook based on event type
  switch (eventType) {
    case 'CONTRACT_CREATION_SUCCEEDED':
      handleContractSuccess(result);
      break;
    case 'TOKEN_TYPE_CREATION_SUCCEEDED':
      handleTokenTypeSuccess(result);
      break;
    case 'TOKEN_CREATION_SUCCEEDED':
      handleTokenMintSuccess(result);
      break;
    case 'CONTRACT_CREATION_FAILED':
    case 'TOKEN_TYPE_CREATION_FAILED':
    case 'TOKEN_CREATION_FAILED':
      handleFailure(eventType, result);
      break;
  }
  
  // Always respond with 200 to acknowledge receipt
  res.status(200).send('OK');
});

function handleContractSuccess(result) {
  console.log(`Contract deployed: ${result.address}`);
  // Update your database, notify users, etc.
}

function handleTokenTypeSuccess(result) {
  console.log(`Token-type created: ${result.tokenTypeId}`);
  // Update your database, notify users, etc.
}

function handleTokenMintSuccess(result) {
  console.log(`Token minted: ${result.tokenId}`);
  // Update your database, notify users, etc.
}

function handleFailure(eventType, result) {
  console.error(`Failed event: ${eventType}`);
  console.error(`Error: ${result.errorMessage}`);
  // Handle failure, retry logic, user notification, etc.
}

app.listen(3000, () => {
  console.log('Webhook server running on port 3000');
});
```

```python Python/Flask
from flask import Flask, request, jsonify
import logging

app = Flask(__name__)
logging.basicConfig(level=logging.INFO)

@app.route('/webhooks/nft-api', methods=['POST'])
def handle_webhook():
    data = request.get_json()
    event_type = data.get('eventType')
    result = data.get('result')
    
    logging.info(f"Received webhook: {event_type}")
    logging.info(f"Status: {result.get('status')}")
    
    # Process the webhook based on event type
    if event_type == 'CONTRACT_CREATION_SUCCEEDED':
        handle_contract_success(result)
    elif event_type == 'TOKEN_TYPE_CREATION_SUCCEEDED':
        handle_token_type_success(result)
    elif event_type == 'TOKEN_CREATION_SUCCEEDED':
        handle_token_mint_success(result)
    elif event_type in ['CONTRACT_CREATION_FAILED', 'TOKEN_TYPE_CREATION_FAILED', 'TOKEN_CREATION_FAILED']:
        handle_failure(event_type, result)
    
    # Always respond with 200 to acknowledge receipt
    return jsonify({'status': 'success'}), 200
```

## Best Practices

<AccordionGroup>
  <Accordion title="Endpoint Security" icon="shield">
    **Security Recommendations:**
    - Use HTTPS for your webhook endpoints
    - Implement proper authentication verification
    - Validate webhook signatures if provided
    - Rate limit your webhook endpoints
    - Log all webhook events for debugging
  </Accordion>
  
  <Accordion title="Error Handling" icon="exclamation-triangle">
    **Robust Error Handling:**
    - Always respond with HTTP 200 for successful processing
    - Implement idempotency to handle duplicate webhooks
    - Use try-catch blocks around webhook processing
    - Log errors for debugging and monitoring

## Troubleshooting

<AccordionGroup>
  <Accordion title="Webhook not receiving notifications">
    **Possible causes:**
    - Endpoint URL is not accessible
    - Authentication configuration is incorrect
    - Firewall blocking incoming requests
    - Server is down or responding with errors
    
    **Solution:** Verify your endpoint is accessible, check authentication settings, and ensure your server responds with HTTP 200.
  </Accordion>
  
  <Accordion title="Duplicate webhook notifications">
    **Possible causes:**
    - Network issues causing retries
    - Server responding with non-200 status codes
    - Processing taking too long
    
    **Solution:** Implement idempotency checks using the event ID and ensure quick response times.
  </Accordion>
  
  <Accordion title="Missing webhook events">
    **Possible causes:**
    - Endpoint was down during event
    - All retry attempts failed
    - Webhook configuration was disabled
    
    **Solution:** Use the status check endpoints as backup and ensure high availability for your webhook endpoints.
  </Accordion>
</AccordionGroup>
